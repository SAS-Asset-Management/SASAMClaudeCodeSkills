Core Concepts

FMEA

Identifies failure modes, their effects on system performance, and existing controls. Uses Risk Priority Number (RPN = Severity × Occurrence × Detection) to prioritise action.

FMECA

Extends FMEA with formal criticality assessment — either qualitative (criticality ranking matrix) or quantitative (criticality number per MIL-STD-1629A). Standard in defence and heavy industry.

The Hierarchy

Every FMEA/FMECA is built on a consistent decomposition. This hierarchy is non-negotiable — every analysis must trace from system context down to failure modes.

System (e.g. Tram Braking System)
  → Sub-system (e.g. Friction Brake)
      → Component / Maintainable Item (e.g. Brake Pad Assembly)
          → Function (e.g. Convert kinetic energy to heat via friction)
              → Functional Failure (e.g. Unable to generate braking force)
                  → Failure Mode (e.g. Brake pad worn below minimum thickness)
                      → Failure Effect (local / system / end effect)
                      → Failure Cause/Mechanism (e.g. Normal abrasive wear)

Gap detection: If a user provides data that skips levels in this hierarchy, flag the gaps. Incomplete hierarchies produce incomplete analyses.

Key Distinctions

Concept AvsConcept BHow to Tell Them Apart
Failure mode vs Failure cause Mode = observable manner (“bearing seized”). Cause = why (“loss of lubrication due to seal degradation”). Many poor FMEAs conflate these.
Local effect vs End effect Local = at the component. System = at the sub-system. End = what operator/customer/public experiences. Effects cascade upward.
Evident failure vs Hidden failure Evident = apparent to operators under normal conditions. Hidden = only apparent when demanded (typically protective devices).

Discovery Interview

Before starting any workflow, conduct a discovery interview tailored to the workflow depth.

Essential Questions (All Workflows)

#QuestionDetails
1 What asset/system? Equipment type, make/model, operating context (where, how, how hard), sector
2 Purpose of analysis? Regulatory/compliance, maintenance strategy (RCM), design review (DFMEA), risk assessment, insurance/safety case
3 Which standard? IEC 60812 (default), SAE J1739 (automotive/rolling stock), MIL-STD-1629A (defence), ISO 14224 (petroleum/gas)
4 Available data? Existing FMEAs, asset registers, failure history, OEM manuals, P&IDs, schematics

Supported Standards

StandardBest ForKey Feature
IEC 60812 Broadest applicability — default choice Flexible RPN-based approach, widely recognised
SAE J1739 Automotive, rolling stock, design-focused Separate prevention and detection controls, revised RPN
MIL-STD-1629A Defence, quantitative criticality required Criticality number using failure rate and mode ratio data
ISO 14224 Petroleum, gas, failure taxonomy alignment Standardised equipment and failure classification

Workflow 1: Interpret & Explain

When a user uploads or pastes FMEA/FMECA data and wants to understand it.

  • Parse the input — Identify structure, columns, scoring method, hierarchy depth, standard alignment
  • Summarise the scope — System/asset, failure mode count, scoring method, standard alignment
  • Explain the findings — Highest-risk items, score meanings, hidden failures, common-cause failures
  • Flag quality issues — Missing data, inconsistent scoring, vague descriptions, hierarchy gaps
  • Provide context — Relate findings to operating context and real-world urgency

Lead with the “so what” — what should the reader care about and do next.

Workflow 2: Author from Scratch

Building a new FMEA/FMECA from available data. Follow these 10 steps in sequence.

StepActionQuality Test
1Establish system boundaryWhat’s in scope and out? Interfaces defined?
2Build the hierarchyDecomposed to maintainable item level?
3Define functionsQuantified where possible? Every item has ≥1 function?
4Identify functional failuresTotal loss, partial loss, intermittent, degraded, unintended, over-function?
5Determine failure modesSpecific and observable? Could a technician picture it?
6Describe failure effectsThree levels (local, system, end)? Safety/environmental?
7Identify failure causesDistinct from failure modes? Linked to operating context?
8Assess S, O, D scoresConsistent? Assumptions documented?
9Calculate RPN / criticalityPer selected standard?
10Recommend actionsHigh-priority items addressed? Intervals specified?

Workflow 3: Review & Critique

Score each dimension 1–5 and provide specific findings with examples.

DimensionWhat to Assess
1. Structural completenessHierarchy complete? Functions stated? Orphaned entries?
2. Failure mode qualitySpecific and observable? Common modes covered? Cause-mode confusion?
3. Effects analysis qualityAll three levels? Safety/environmental? Cascading effects?
4. Scoring consistencyScale documented? Similar modes scored consistently? Detection based on current controls?
5. RecommendationsHigh-RPN items have actions? Actions specific? Clear failure-to-task link?
6. Standard alignmentFormat matches standard? Mandatory fields present? Methodology correct?

Review Output Structure

  • Executive summary (overall quality rating, key concerns)
  • Findings by dimension (with specific examples)
  • Priority recommendations (what to fix first)
  • Positive observations (what’s done well)

Workflow 5: Parse Document into Context File

Extract FMECA-relevant information from technical documents into structured markdown files for reuse in future sessions. This is foundational work that dramatically improves subsequent analysis quality.

Supported Document Types

Document TypeWhat to Extract
OEM/Maintenance ManualEquipment hierarchy, functions, specs, recommended maintenance, operating limits, OEM failure modes
Asset Register / BOMHierarchy, naming conventions, parent-child relationships, quantities, criticality tags
Maintenance PlanCurrent PM tasks, intervals, condition monitoring, shutdown scope, maintenance philosophy
P&ID / System DiagramSystem boundaries, equipment arrangement, process flow, instrumentation, protective devices
Inspection / Condition ReportCurrent condition, defects, degradation rates, remaining life estimates
Failure History / Work OrdersActual failure modes, frequency, root causes, downtime, repair actions, repeats
Existing FMEA/FMECAPrevious scope, scoring, identified failure modes, recommendations, gaps
Safety Case / HAZOPHazards, consequences, safeguards, SIL ratings, safety-critical items
Reliability DataFailure rates, MTBF, MTTR, availability, Weibull parameters

Context File Naming

{client}-{asset}-{doctype}-context.md

Examples:
  yarratrams-bclass-oem-context.md
  geelongport-shiploader1-condition-context.md
  santos-compressor-fmea-context.md

Extraction Rules

  • Equipment hierarchy — Decompose to maintainable item level, preserve document numbering
  • Functions — Verb + object + quantified performance, cite source section
  • Operating context — Environment, duty cycle, loading, temperature, utilisation
  • Maintenance context — Tasks, intervals, basis (OEM/regulatory/experience)
  • Failure information — Record exactly as described, map to taxonomy, preserve quantitative data
  • Protective devices — List every device, what it protects, test interval, last test result

Protective devices are the most commonly missed items in FMEAs. Extracting them upfront during document parsing is high-value work that prevents gaps in the analysis.

Supported Sectors

The skill includes sector-specific failure mode taxonomies that serve as starting points for analysis. Always read the relevant taxonomy before authoring.

Rolling Stock

Rail, tram, and train assets. Covers traction, braking, doors, HVAC, bogies, couplers, and signalling interfaces.

Ports & Marine

Wharves, ship loaders, cranes, conveyor systems, fenders, and marine navigation aids.

Mining & Resources

Crushers, mills, conveyors, processing plants, heavy mobile equipment, and dewatering systems.

General Industrial

Cross-sector: pumps, motors, valves, HVAC, electrical distribution, civil structures, and instrumentation.

Common Pitfalls to Flag

#PitfallWhat to Look For
1Vague failure modes“Component fails” is not a failure mode
2Cause-mode confusion“Corrosion” is usually a cause, not a mode
3Inconsistent scoringSame failure type scored differently without justification
4Missing hidden failuresProtective devices frequently missed
5Single-point-of-failure blindnessNot identifying absent redundancy
6Copy-paste syndromeGeneric descriptions reused without adaptation
7Detection score confusionRates current controls, not theoretical capability
8Severity creepEverything rated 9–10 makes analysis useless for prioritisation
9Missing operating contextSame equipment, different environment = different failure behaviour

When data is insufficient: Be transparent. State assumptions clearly. Flag where engineering judgement has been applied vs data-supported assessment. Never fabricate failure rate data — use qualitative ranges and state the basis.
