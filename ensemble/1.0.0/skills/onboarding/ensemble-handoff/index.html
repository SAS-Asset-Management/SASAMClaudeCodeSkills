The one idea

You hand work to a fleet of agents — over GitHub.

No SSH, no servers, no dashboards. A repo per engagement is the whole control surface. You write a task; the Ensemble does it; you review a pull request.

Control plane

GitHub. Packets in, results out as PRs. Repo access is the permission wall.

Data plane

MinIO + Git-LFS over Tailscale. Big files ride the mesh, not GitHub.

The fleet

agnarBergstrom (conductor) + 14 specialists on the R740.

Transition: "It looks like normal git — because it is." The magic is who's on the other end.

M0
The "0"

Install Claude Code & the toolbelt

Three things: the CLI, the git toolbelt, and the Ensemble skills. ~10 minutes, once.

1 · Claude Code

Sign in on your Max plan, install the CLI, run claude and log in.

2 · Toolbelt

git, gh (then gh auth login), git-lfs, and join the Tailscale tailnet.

3 · Skills

Install the SASAMClaudeCodeSkills plugin — the /tether…/status skills.

macOS: Homebrew for everything. Linux: the official install scripts. Tailscale is what lets you pull big files later — do it now.

Verify your install

# each should print a version / a logged-in status claude --version gh auth status git lfs version tailscale status
  • claude logged in on your Max plan
  • gh authenticated to GitHub
  • git-lfs installed; Tailscale connected
  • SASAMClaudeCodeSkills plugin installed
If any line errors, fix it before moving on — the later modules assume all four work.

Mental model

How the pieces fit

ThingWhat it is
queue branchThe mailbox — your packets go to handoffs/inbox/; the poller claims them.
main branchThe reviewed record — results land in handoffs/outbox/<id>/ via PR. Protected.
Routingkind: project → larsFrederickson; anything else → agnarBergstrom.
YouWrite the task. You never pick a specialist — the conductor does.
The single most freeing idea: you describe the work and its done-criteria. Who does it is the Ensemble's problem, not yours.

Review tiers — who must approve

auto

Low-stakes. Auto-merges on green checks.

light

Any engagement consultant approves.

full (default)

The requesting consultant approves.

founder

Scriv approves (CODEOWNERS).

  • I know queue vs main, project→Lars / else→Agnar, and the four tiers
Tier is a field on the packet. It decides whose approval turns the PR's tier-gate check green.
M2
Your five commands

The whole workflow is five verbs

CommandWhat it does
/tether <uuid|name>Bind to an engagement: resolve it in the registry, clone or add the ensemble remote, fetch main+queue, orient you.
/handoffWrite a task packet and fast-forward push it to queue — your only direct push.
/syncFour-part report: results to collect · PRs awaiting you · returned packets · claims in flight.
/collect [id]Pull main, fetch + sha256-verify LFS blobs, summarise results.
/statusRegistry-wide view + the poller's last heartbeat.
There's also a SessionStart hook: open a tethered repo and /sync runs automatically before your first prompt.

The everyday loop

# once per engagement /tether transdev-iso55001 # whenever you have work /handoff # the Ensemble polls every ~2 min — later… /sync # what needs me? /collect # pull the finished results
  • I can name all five commands and when to use each
Muscle memory: tether once, then handoff → sync → collect for the life of the engagement.
M3
Writing a good packet

The packet is a brief for an agent with no memory

--- id: HX-2026-0610-tram-fmeca engagement: transdev-iso55001 requested_by: jonathan kind: project # → larsFrederickson review_tier: full # you must approve route_hint: api deadline: 2026-06-20T17:00+10:00 inputs: [data/asset-register-trams.csv] definition_of_done: - Re-score all 11 assets against rubric v3 (S×O×D) - Flag the top-5 by RPN with a maintenance tactic --- ## Brief Re-score Transdev's tram subsystem register…
The definition_of_done IS the contract. Write it as verbatim, testable acceptance criteria — the agent has zero session context.

Two rules that save you pain

Done-criteria are testable

"Re-score all 11 assets", not "look at the register". The reviewer checks each line.

Big files use LFS

Inputs >10 MB must be Git-LFS pointers or live under data/. /handoff stops you otherwise.

  • My packet has a testable definition_of_done and LFS-safe inputs
A vague done-list is the #1 cause of a rejected result. Spend the extra minute here.
M4
From a client site

What works on hotel wifi

GitHub only → control plane

Tether, handoff, sync, review, merge — all work with nothing but GitHub access.

Tailscale → data plane

Only /collect's big blobs need the mesh. Off-mesh, it tells you what it skipped.

  • No secrets, credentials, or client PII in any file or packet
  • Australian English in everything client-facing
You can run the whole control plane from a client boardroom. Blobs just wait until you're back on the tailnet.
M5
Review & collect

The pull request is the gate

The Ensemble opens a PR into main. The tier-gate check stays red until the right person approves — for a full packet, that's you.

# see what's waiting, then pull the merged result /sync /collect HX-2026-0610-tram-fmeca # sha256-verifies LFS blobs
  • I approve as the required reviewer; red → green; it merges
  • Results land in handoffs/outbox/<id>/ on main
Approving is a normal GitHub review. tier-gate re-checks the moment you approve and flips green.
M6
The "100"

Hands-on: run a real handoff

# against the practice engagement /tether sasam-onboarding-sandbox /handoff # a tiny sample task, kind: task, tier: full # wait ~2 min — the poller claims it, opens a PR /sync # your PR is awaiting review # approve the PR on GitHub → it merges /collect
This is the graduation exercise. When /collect shows your result summary, you've gone end to end.

You've reached 100 when…

  • Tethered the sandbox and pushed a packet to queue
  • Watched the poller claim it and open a PR
  • Approved the PR; tier-gate went green; it merged
  • /collect pulled your result and summarised it
All four ticked = you're operational. Real engagements work exactly the same way.
M7
When something's odd

Troubleshooting & FAQ

SymptomWhat's happening / do
"fast-forward push failed"The queue advanced — re-run /handoff; it rebases onto the new tip.
Packet went quietA claim can stall; the janitor returns it to inbox and bumps retries. Check /status.
"LFS mesh unreachable"You're off Tailscale. Control plane still works; /collect skips blobs and lists them.
Is the fleet alive?/status shows the poller's last heartbeat and per-repo claims.
Most "stuck" feelings are just the 2-minute poll cadence. /status is your single source of truth.
