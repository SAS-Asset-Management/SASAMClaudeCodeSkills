Step 1 · Open a terminal

The "terminal" is a text window where you type commands. Opening it:

macOS press ⌘ Space, type Terminal, hit Return Windows Start → type Terminal Linux Ctrl Alt T

You'll see a window with a blinking cursor after a $ (or %). That $ is the prompt — it means "type here". You type a command and press Return.

Terminal — zsh
$ # this is the prompt. you type after the $ and press Return.
$

Throughout this deck, a line starting with $ is something you type; the dimmed lines under it are what the computer prints back.

This is the single biggest unlock for a true beginner — what the $ is and that dimmed = output. Don't rush it.

Step 2 · Homebrew (Mac only — Win/Linux skip)

Homebrew lets you install the other tools with one short command each. Paste this whole line into the terminal and press Return:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

↑ click anywhere on the box to copy it to your clipboard

Terminal — zsh
$ /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
==> Checking for `sudo` access (which may request your password)...
==> Installation successful!
$ brew --version
Homebrew 4.3.8
Windows Homebrew not needed — use winget (built in)Linux use apt (built in)

It may ask for your Mac password (typing shows nothing — that's normal) and take a few minutes. ✅ Success = brew --version prints a number.

Two beginner gotchas to call out: the password prompt shows no characters, and it's slow. Both are normal.

Step 6 · git-lfs

Large File Storage — keeps big inputs (spreadsheets, point clouds, images) out of the way. Install, then run git lfs install once:

✅ Success = "Git LFS initialized." You only run git lfs install once per machine.

Quick — one extra "install" sub-step (git lfs install) beyond the brew install. That's the only nuance.

M1
How it works

You hand work to a fleet of AI agents — over GitHub

You describe a task and its done-criteria. The Ensemble (a conductor agent + 14 specialists on our server) does it, and hands you back a pull request to review. You never pick which agent — the conductor decides.

GitHub = the inbox/outbox

Tasks go in, finished work comes back. Repo access is the wall that keeps engagements separate.

Two lanes

queue = your outbox for tasks. main = the reviewed record of finished work.

Routing

A project goes to Lars (plans + delegates). Anything else goes to Agnar (triage).

The freeing idea: describe the work + done-criteria; who does it is the Ensemble's problem.

Three GitHub words you'll meet

WordPlain English
BranchA safe copy to work on without touching the finished version (main).
Pull request (PR)"Here's my work — please review & approve." It's the review screen you'll click through.
MergeApproving the PR — the work joins main (becomes official).

Review tiers — whose approval is needed

auto

Low-stakes — merges itself on green checks.

light

Any consultant approves.

full

The requester approves.

founder

Shane approves.

  • I get branch / PR / merge, the queue→main lanes, and the four tiers
Define PR/branch/merge before any module uses them. The tier table maps directly to who clicks approve.

M3
Writing a task

A task is a brief for someone with no memory

Run /handoff and answer a few questions. The most important is the definition of done — write it as clear, testable bullet points, because the agent has zero prior context.

claude — sasam-transurban-wcx-ncx
> /handoff
What's the brief?      Re-score the tram asset register against rubric v3
Inputs?               data/asset-register.csv
Done when…?           All 11 assets re-scored; top-5 by RPN flagged
Kind / tier?          project / full
✓ Packet HX-2026-0610-fmeca pushed. Norbert polls every ~2 min.

Defaults are sensible (kind: project, tier: full) — just press Return to accept. A "packet" is simply your task, saved with an ID.

The DoD discipline is the #1 quality lever — testable bullets, written for zero context.

Two rules that save you pain

Done-criteria are testable

"Re-score all 11 assets" — not "look at the register". The reviewer checks each line you wrote.

Big files use LFS

Inputs over 10 MB go in the data/ folder (git-lfs handles them). /handoff warns you if one's too big.

  • My task has testable done-criteria and LFS-safe inputs
A vague done-list is the top cause of a rejected result. One extra minute here pays off.

What works on hotel wi-fi

GitHub-only → it all works

Tether, handoff, sync, review, approve — with nothing but internet access.

Tailscale → just the big files

Only /collect's large downloads need the private network. Off it, it tells you what it skipped.

M5
Review & collect

The pull request is where you approve

When the fleet finishes, it opens a PR. A check called tier-gate stays red until the right person approves — for a full task, that's you. Approve it and it goes green and merges.

claude — sasam-transurban-wcx-ncx
> /sync
1 PR waiting on you: #7 “FMECA re-score” — needs your approval
   → approve it on GitHub, then…
> /collect
✓ Pulled FMECA worksheet + summary (sha256 verified).
  • I approve as the required reviewer; the check goes red → green; it merges
  • Results land in the engagement's outbox and /collect brings them down
Approving is a normal GitHub review (a green button). tier-gate re-checks the instant you approve.

M7
When something's odd

Troubleshooting & FAQ

You see…What's happening / do
"command not found"That tool didn't install — re-run its install step from M0.
"fast-forward push failed"The queue moved on — just run /handoff again; it sorts itself out.
It's gone quietThe fleet polls every ~2 minutes. Run /status to see what's in flight.
"LFS / mesh unreachable"You're off Tailscale — everything still works; big files just wait.
Stuck on a stepAsk Claude Code in plain English ("what does this error mean?") or ping Shane.
Most "stuck" feelings are the 2-minute poll. /status is the source of truth. And Claude Code itself can explain any error.
